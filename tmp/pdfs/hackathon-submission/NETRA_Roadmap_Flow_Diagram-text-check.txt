1. Roadmap Purpose And Decision Model
This document converts NETRA's current prototype status into an evidence-based implementation plan. It is not a
feature wish list. Every phase has a defined objective, bounded deliverables, measurable verification criteria,
dependencies, and an exit gate. A phase is complete only when its evidence is reproducible.

Guiding principles
- Preserve the reliable stored-PCAP workflow while hardening advanced paths.
 
- Do not promote local benchmark metrics into general accuracy claims.
 
- Treat human review as part of the detection method.
 
- Require security, integrity, usability, performance, and governance evidence.
 
- Record unresolved risk instead of hiding it behind implementation completeness.
 
2. Validated Baseline And Evidence Flow
The primary validated workflow begins with an authorized PCAP or PCAPNG file. The backend validates the
evidence, calculates a SHA-256 identity, encrypts the long-term artifact, invokes TShark and Zeek, reconstructs
sessions, derives features, applies rules and anomaly scoring, persists case records, and produces reports, exports,
integrity results, and custody history [CLM-001 to CLM-006].

4. Twelve-Month Phased Roadmap
 Figure R3. Phased implementation roadmap with explicit evidence gates.
Phase 0 - Current validated baseline
Objective: Preserve the reliable demonstration path and freeze the factual baseline.
Deliverables: claims matrix, reproducible build, stored-PCAP workflow, report/export artifacts, current benchmark
record, production blockers, and known-limitations register.
Exit evidence: submission package reproduces from source; current core validations pass; no unsupported claim
remains.
Phase 1 - 0 to 30 days: security and reliability baseline
Objective: Remove preventable release risks and stabilize advanced ingestion paths.
Deliverables: secret rotation, production debug/host/origin/HTTPS settings, RLS/RBAC inventory, storage-policy
review, signed integrations, malformed-evidence tests, unauthorized-access tests, and repeatable replay finalization.
Dependencies: access to the managed data-plane configuration, supported Windows/Linux capture environments,
and a designated security reviewer.
Exit gate: all required deployment checks pass in a controlled environment; negative authorization tests pass; replay
completes repeatedly without integrity or custody failure.

Phase 2 - 1 to 3 months: analytical quality
Objective: Replace demonstration metrics with a defensible evaluation method.
Deliverables: broader labeled corpus, benign and ambiguous cases, external samples, independent
train/validation/test partitions, per-class threshold tuning, calibration, model card, rule documentation, error
taxonomy, and analyst disposition records.
Dependencies: lawful access to representative data, labeling protocol, data governance, and compute/storage
budget.
Exit gate: benchmark method is reproducible; per-class metrics and false-positive causes are published; model
limitations are explicit; no critical RLS/RBAC defect remains.
Phase 3 - 3 to 6 months: controlled pilot readiness
Objective: Establish that the system can operate reliably and support investigator tasks.
Deliverables: large-PCAP workload tiers, concurrent-case benchmarks, worker retry and dead-letter recovery,
backup/restore tests, task-based usability protocol, controlled practitioner sessions, explanation revisions, and
report-template revisions.
Dependencies: pilot authorization, representative hardware, test captures, operational monitoring, and research
consent/process approval.
Exit gate: target workloads complete within documented limits; injected failures recover without evidence
corruption; pilot users complete core tasks without critical errors.
Phase 4 - 6 to 12 months: operational approval
Objective: Decide whether evidence supports a bounded production deployment.
Deliverables: monitored SIEM/webhook operations, incident-response drill, retention/legal-hold procedure,
key-management procedure, release and rollback runbook, authority-led legal-process review, and final residual-risk
register.
Dependencies: operating organization, legal/forensic authority, production infrastructure, monitoring, support
ownership, and approved release scope.
Exit gate: security, operational, analytical-quality, usability, and legal-process owners approve a documented scope.
Otherwise, deployment remains gated.

Final roadmap position
The roadmap does not assume that twelve months automatically produces a production system. Time boxes organize
work; evidence gates determine progression. NETRA should advance only when each phase demonstrates
acceptable integrity, analytical quality, security, reliability, usability, and governance for a clearly bounded
operational scope.
