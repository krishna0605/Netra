Abstract
NETRA is an evidence-first network and packet forensics prototype designed for authorized cybercrime
investigation workflows. The project addresses an integration problem rather than a missing packet-analysis
capability. Mature tools already dissect packets, generate network-security logs, detect suspicious patterns, and
centralize events. The remaining challenge is to preserve evidence identity and context while moving from a capture
file to reviewable findings, case history, reports, exports, and custody records. NETRA combines a React
investigation console, a Django trusted-processing boundary, TShark and Zeek analysis, rule-based detection, a
small explainable Random Forest prototype, PostgreSQL case persistence, private artifact storage, integrity
verification, and a hash-linked custody ledger.
This report evaluates NETRA as an engineering prototype. Repository evidence supports the stored PCAP/PCAPNG
workflow, packet-tool availability, case-scoped analysis, selected report/export flows, integrity checks, and custody
verification [CLM-001 to CLM-006]. A six-case detection benchmark produced precision 0.625, recall 1.0, and F1
0.7692, with three false positives [CLM-008]. The anomaly model has only six training rows [CLM-007]. These
results demonstrate implementation progress but do not establish field accuracy, production readiness, or legal
admissibility. The proposed twelve-month roadmap therefore prioritizes security hardening, broader evaluation,
scalability, investigator research, operational pilots, and authority review.
Keywords: network forensics, packet capture, PCAP, digital evidence, incident response, explainable anomaly
detection, chain of custody, cybercrime investigation.
1. Introduction
1.1 Background
Network traffic can provide evidence of scanning, credential attacks, command-and-control behavior, suspicious
transfers, lateral movement, tunneling, and other security-relevant activity. However, a packet capture is not
automatically an investigation. Raw frames must be validated, preserved, decoded, grouped into sessions,
interpreted, connected to case context, and converted into artifacts that another reviewer can understand.
NIST guidance frames incident response and digital forensics as lifecycle activities rather than isolated tool
operations [R1, R2]. RFC 3227 similarly emphasizes disciplined evidence collection and archiving [R4]. These
principles imply that a useful investigation platform should preserve relationships among the source evidence,
derived records, analyst decisions, reports, and access history.
1.2 Problem statement
Existing tools are strong but specialized. TShark provides detailed packet dissection and structured output [R5].
Zeek generates rich records describing observed network behavior [R6]. Suricata provides IDS, IPS, and
network-security monitoring [R7]. SIEM platforms aggregate events, while MITRE ATT&CK identifies network
traffic as a data source for detecting adversary behavior [R8]. In practice, the outputs from these tools can still
require manual correlation, note-taking, screenshots, spreadsheet work, and report assembly.
The central problem addressed by NETRA is therefore:
How can an authorized investigator move from stored network evidence to a traceable, explainable,
and reviewable case package without losing evidence identity or hiding analytical uncertainty?

1.3 Proposed contribution
NETRA proposes an integrated evidence-to-report workflow with five contributions:
1. Evidence continuity: validation, identity hashing, encrypted storage, manifests, derived findings, reports, and
 custody events remain linked to a case.
2. Tool orchestration: TShark and Zeek are used as established analysis engines rather than reimplemented.
 
3. Reviewable detection: deterministic rules and anomaly features produce evidence-linked hypotheses for human
 review.
4. Investigator-oriented outputs: packet/session evidence, graphs, timelines, reports, exports, and integrity state are
 organized around investigative tasks.
5. Explicit maturity boundaries: validated, partial, experimental, and future capabilities are separated.

3. Existing Technology Landscape
3.1 Packet inspection
TShark is the command-line component of Wireshark. Its documentation covers capture-file input, protocol
dissection, display filters, field selection, and structured output [R5]. This makes it suitable for repeatable
server-side extraction. NETRA does not claim to replace TShark; it invokes it within a controlled evidence
workflow.
3.2 Network-security monitoring
Zeek is designed to produce detailed records about network activity [R6]. Those records complement packet-level
output by providing protocol and service context. NETRA uses offline Zeek analysis when available and records
tolerant failure rather than making the entire workflow dependent on a single tool.
3.3 Intrusion detection and security monitoring
Suricata combines IDS, IPS, and network-security monitoring functions [R7]. SIEM and log-management platforms
aggregate security events and support retention, search, and operational review [R3]. These platforms are important
integration targets. NETRA's narrower role is to build a case package around authorized network evidence and
investigator decisions.
3.4 Detection knowledge and datasets
MITRE ATT&CK identifies network traffic as a useful detection data source [R8]. CICIDS2017 provides labeled
benign and attack traffic for intrusion-detection research [R9, R10]. These resources support a systematic detection
and evaluation program, but they do not eliminate the need for environment-specific baselines, independent samples,
and analyst validation.

4. Existing Gaps
The fragmented workflow below is a conceptual synthesis. It does not assert that every organization uses the same
tools or experiences the same severity. It shows where evidence context can be lost when specialized outputs are
transferred manually.
 Figure 1. Conceptual existing workflow and points where evidence context can become fragmented.
4.1 Gap G1: fragmented evidence context
Packet fields, Zeek records, IDS alerts, analyst notes, and report statements may use different identifiers and
schemas. Without a common case and evidence model, reviewers must reconstruct how a conclusion relates to the
original capture.
4.2 Gap G2: manual correlation and reporting
Manual transfer increases effort and can produce inconsistent terminology, missing timestamps, detached
screenshots, or conclusions without clear supporting records. NIST's lifecycle guidance supports retaining analysis
and reporting context together [R1, R2].
4.3 Gap G3: weakly contextualized automated findings
An alert label or anomaly score is not sufficient for forensic interpretation. Reviewers need the triggering behavior,
contributing features, related packets/sessions, confidence basis, and alternative explanations. Explainability can
help expose feature contributions, but explanation cannot compensate for inadequate training data [R11, R12].

4.4 Gap G4: separate integrity and custody processes
Evidence hashes, access records, report artifacts, and custody notes are often maintained outside the analysis view.
This separation makes it harder to demonstrate continuity between an original file and a derived conclusion.
4.5 Gap G5: maturity claims without explicit gates
A prototype can appear complete while still lacking representative accuracy tests, security-policy review, workload
benchmarks, pilot evidence, or legal approval. NETRA therefore uses gated language and a claims matrix instead of
treating implementation as proof of operational readiness.

5.1 Design principles
- Evidence before visualization: no case records should imply analysis before authorized evidence is processed.
 
- Derived data remains attributable: every finding should retain case, evidence, packet/session, method, and
 timestamp context.
- Human review remains authoritative: automated outputs are hypotheses or prioritization aids.
 
- Limitations are visible: no TLS decryption, no certified attribution, no production or legal claim without review.
 
- Security boundaries are explicit: raw evidence and privileged storage actions remain behind the backend.
 
6. Functional Requirements
6.1 Evidence intake
The system shall accept authorized PCAP and PCAPNG files, validate extension and structure, enforce size policy,
calculate an identity hash, create case/evidence records, and reject mismatched or malformed input before long-term
storage [CLM-001, CLM-003].
6.2 Analysis and reconstruction
The system shall invoke packet-analysis tools, extract packet and protocol fields, produce Zeek summaries where
available, reconstruct sessions, calculate features, and persist derived records with evidence linkage [CLM-002,
CLM-004].
6.3 Detection and prioritization
The system shall apply deterministic rules and explainable anomaly scoring, record method and confidence, connect
findings to evidence, and support investigator disposition. Detection output shall not be described as proof of
compromise without corroboration.
6.4 Investigation views
The interface shall provide a case overview, suspicious-activity review, traffic evidence, graph/timeline context,
report/export actions, integrity state, and custody history. Empty states shall not fabricate demonstration records.
6.5 Reporting and interoperability
The system shall generate reviewable report and export artifacts, retain artifact metadata, log access, and support
basic SIEM-oriented export. Format claims must identify which flows have been validated [CLM-005].
6.6 Governance
The system shall authenticate users in managed mode, keep privileged credentials on the backend, maintain
audit/custody records, and expose production blockers through a readiness check [CLM-012, CLM-013].

8. System Architecture
The proposed architecture separates the investigator experience, trusted evidence processing, analysis tools, and
managed data plane.

 Figure 3. Proposed NETRA logical architecture and major trust boundaries.
8.1 Frontend
The React and TypeScript console organizes investigative tasks. It does not receive service-role credentials or
directly manipulate private evidence objects. In managed mode it carries a user identity token to the backend.
8.2 Backend trusted-processing boundary
Django validates requests and evidence, controls temporary plaintext processing, invokes native tools, applies
analysis logic, writes case records, generates artifacts, and mediates downloads. This boundary reduces the risk of
exposing raw evidence or privileged storage operations to browser code.
8.3 Analysis engines
TShark provides packet dissection and field extraction. Zeek provides structured records. Rule logic evaluates
explicit behavior patterns. The anomaly layer derives features and produces a supplementary risk hypothesis. Tool
errors are recorded; Zeek failure does not silently become a successful Zeek result.
8.4 Data plane
PostgreSQL stores case and investigation records. Private object storage contains encrypted evidence and generated
artifacts. PGMQ supports optional worker coordination, while realtime events and polling provide operational status
[CLM-013].

9. Evidence Processing Lifecycle
 Figure 4. Sequence from authorized evidence upload to report, export, and integrity verification.
9.1 Intake and validation
Validation occurs before analysis and long-term storage. The file extension, size, header or magic bytes, declared
evidence type, and case context are checked. Rejected input should produce a clear error without creating misleading
analysis records.
9.2 Identity, encryption, and storage
The plaintext SHA-256 hash establishes evidence identity. The long-term artifact is encrypted and written to private
storage. The manifest records relevant identifiers and storage hashes. Encryption does not remove the need for
access control, key governance, retention policy, and audit.
9.3 Temporary analysis
Native analysis tools require an authorized working copy. The design goal is to limit the lifetime and exposure of
plaintext data, remove temporary material after processing, and prevent it from appearing in logs or reports.
9.4 Derived records
Packets, sessions, protocol summaries, features, alerts, anomaly records, graph edges, timeline entries, and report
previews are linked to the case and evidence. This linkage is central to traceability.

9.5 Artifact generation and verification
Reports and exports are generated from persisted case evidence, encrypted before storage, and recorded as artifacts.
Integrity verification recalculates expected hashes and validates custody continuity [CLM-005, CLM-006].
10. Detection And Explainable Anomaly Method
 Figure 5. Hybrid detection method with explicit human review and feedback paths.
10.1 Feature families
The prototype extracts features related to internal/external hosts, unique ports, destination and port fan-out, session
volume, DNS length and repetition, ICMP size, beacon pairs, SSH connections, query counts, and risk hints. These
features represent observable metadata and traffic behavior.
10.2 Deterministic rules
Rules cover credential brute force, scanning, remote execution indicators, suspicious SMTP transfer, lateral
movement indicators, beaconing, tunneling, and exfiltration hypotheses. Each rule should document its data
requirements, threshold, confidence basis, known confounders, and recommended corroboration.

10.3 Random Forest prototype
The repository contains a RandomForestClassifier artifact trained on six local rows [CLM-007]. This is sufficient to
demonstrate the code path, model persistence, and explanation interface. It is not sufficient for a claim of
generalizable accuracy. The recorded perfect model metrics reflect the tiny local training sample and must not be
used as headline performance evidence.
10.4 Human review
Human review is part of the method rather than a post-processing convenience. An investigator may confirm a
finding, reject it, request more context, or correlate endpoint/server logs. Feedback should later support threshold
tuning and dataset curation without overwriting the original automated result.
10.5 Encrypted traffic boundary
NETRA does not decrypt TLS payloads [CLM-009]. It can use endpoints, ports, timing, sizes, SNI/certificate fields
where present, DNS behavior, session structure, and other metadata. It cannot infer protected content with certainty
or prove account compromise without corroborating sources.

12. Security, Privacy, Integrity, And Custody
 Figure 6. Evidence identity, encrypted storage, custody events, and governance controls.
12.1 Security controls
The intended controls include managed authentication, backend-only privileged credentials, private storage,
encrypted artifacts, explicit allowed hosts/origins, disabled debug mode, signed integrations, key rotation, and
least-privilege policies. The current deployment readiness remains blocked, which is the correct status until required
checks pass [CLM-012].
12.2 Privacy controls
Packet captures may contain personal, confidential, or legally restricted information. NETRA must be used only
with authorized evidence. Screenshots and reports should minimize payload exposure. Retention, legal hold, export,
deletion, and key-preservation policies require organizational approval.
12.3 Integrity and custody
The prototype records hashes, manifests, integrity checks, and hash-linked custody events [CLM-003, CLM-006].
The technical objective is to detect modification and provide a reviewable handling history. This supports legal
review but does not itself establish admissibility [CLM-014].

12.4 Legal boundary
Admissibility depends on jurisdiction, acquisition authority, operating procedure, examiner competence, tool
validation, documentation, and testimony. NETRA should therefore produce transparent technical records and defer
legal conclusions to the relevant authority.

movement, and suspicious SMTP cases produced expected findings, while additional beaconing, scan, and SMTP
predictions contributed false positives.

15. Evaluation Framework
The next evaluation phase must extend beyond detection metrics.
 Figure 7. Multi-dimensional evaluation framework for NETRA.
15.1 Detection validity
Use a versioned corpus with benign, known attack, ambiguous, and environment-specific samples. Separate training,
validation, and test data. Report per-class precision, recall, F1, confusion matrices, calibration, and false-positive
causes. Retain tool, rule, and model versions.
15.2 Forensic integrity
Test manifest generation, artifact hashing, tamper detection, custody-chain verification, report reproducibility,
export reproducibility, key rotation, backup/restore, and evidence recovery.
15.3 Usability
Conduct task-based studies with intended users. Measure task completion, time-on-task, navigation errors, missed
evidence, explanation comprehension, report completeness, and confidence. Compare the integrated workflow with
the participant's existing process.

17. Implementation Roadmap
 Figure 8. Twelve-month roadmap with evidence-based exit gates.
17.1 Phase 0: current baseline
Preserve the stored-PCAP workflow as the reliable demonstration path. Freeze the evidence schema and claims
matrix for submission. Record current detection and deployment limitations.
17.2 Phase 1: 0-30 days - security and reliability baseline
- Rotate exposed or development credentials.
 
- Enforce production debug, host, origin, HTTPS, and signing settings.
 
- Inventory RLS/RBAC and storage policies.

- Add negative tests for malformed evidence and unauthorized access.
 
Exit gate: required security readiness checks pass in a controlled environment, and replay completes repeatably
with integrity preserved.
17.3 Phase 2: 1-3 months - repeatable analytical quality
- Expand the labeled corpus with benign, attack, and ambiguous cases.
 
- Establish independent train/validation/test partitions.

18. Expected Outcomes
If the proposed workflow is validated, NETRA may reduce avoidable context switching, improve the traceability of
technical findings, produce more consistent case artifacts, and strengthen evidence-handling discipline. These are
hypotheses, not current measured outcomes.
The project should measure success using:
- Median time to move from authorized upload to reviewable case summary.
 
- Percentage of findings with complete evidence and method provenance.
 
- Analyst task-completion and explanation-comprehension rates.
 
- Per-class detection precision, recall, and false-positive rate.
 
- Integrity and custody verification success rate.
 
- Large-capture completion, recovery, and artifact-generation performance.
 
- Number and severity of unresolved security or policy findings.

19. Conclusion
NETRA demonstrates a credible engineering direction for evidence-first network forensics. Its strongest
contribution is not a novel decoder or a certified AI detector. It is the integration of evidence validation, established
analysis tools, case-scoped records, transparent findings, reports, integrity checks, and custody history into one
workflow.
The current prototype is materially implemented but not production complete. Its benchmark corpus is small, the
ML model is experimental, scale is not established, production security remains gated, and legal admissibility
requires authority review. Treating these limitations explicitly makes the proposal stronger: it provides a testable
roadmap from hackathon prototype to evaluated operational system.
