# NETRA - Google Form Answers

## Synopsis / Abstract

NETRA is an evidence-first network forensics platform for authorized cybercrime investigation teams. Packet captures often contain valuable indicators, but turning low-level traffic into a clear case narrative requires several specialist tools and extensive manual correlation. NETRA provides a guided workflow that validates PCAP or PCAPNG evidence, establishes its SHA-256 identity, encrypts the stored artifact, analyzes traffic with TShark and Zeek, reconstructs sessions, applies behavior rules and explainable anomaly scoring, and presents findings through case-scoped dashboards, reports, exports, and custody records. Its differentiator is not a new packet decoder; it connects proven analysis engines to investigator-oriented evidence handling and reporting. The current prototype validates the primary stored-capture workflow while clearly treating live capture, scale, production security, field accuracy, and legal acceptance as further work. The intended outcome is faster, more traceable review without hiding uncertainty or claiming to decrypt protected traffic.

## Literature Review / Existing Innovation & Technology

Existing technologies solve important parts of network investigation. Wireshark and its TShark command-line tool provide mature packet dissection, filtering, and structured capture-file output [R2]. Zeek acts as a network security monitor and produces detailed activity logs that support protocol and behavior analysis [R3]. IDS platforms such as Suricata focus on intrusion detection, prevention, and network security monitoring [R4], while SIEM and log-management systems centralize events for operational review; NIST SP 800-92 describes the infrastructure and processes required for effective log management [R5]. NIST SP 800-86 also frames digital forensics as an incident-response process spanning collection, examination, analysis, and reporting [R1]. Research datasets such as CICIDS2017 show how labeled benign and attack traffic can support repeatable intrusion-detection evaluation [R6, R7]. The remaining opportunity is a coherent investigator workflow that keeps evidence identity, packet-derived findings, explanations, case history, integrity checks, custody actions, and report artifacts together. This matters when evidence must remain reviewable across technical and supervisory roles. NETRA uses established analysis tools rather than replacing them and adds this evidence-to-report orchestration. Its current small benchmark corpus supports engineering validation only, not production accuracy or legal-certification claims.

## Your Approach to Solve the Problem

NETRA follows an evidence-first pipeline. An authorized investigator starts with a stored PCAP or PCAPNG file and case context. The backend validates the extension, size, and capture header before analysis. It computes a SHA-256 identity hash, encrypts the long-term artifact, and records storage and manifest metadata so later integrity checks can reconnect results to the original evidence. A temporary authorized working copy is processed with TShark for packet and protocol fields and with Zeek for structured network-security evidence. NETRA then reconstructs sessions, extracts timing, endpoint, DNS, service, volume, and protocol features, applies transparent rules, and produces explainable anomaly hypotheses. Every finding remains reviewable by a human investigator. Results are persisted as case-scoped packets, sessions, alerts, anomalies, graph edges, timeline points, reports, exports, and custody events. The React interface presents the workflow in investigator language, while Django remains the trusted boundary for raw evidence, tools, storage access, and artifact generation. Metadata can support encrypted-traffic analysis, but NETRA does not decrypt TLS payloads. Stored-capture analysis is the primary validated path; replay, native capture, broader accuracy evaluation, production security policy, scalability, and legal review are staged roadmap items.

## Tools & Technologies to be Used

NETRA uses React, TypeScript, Vite, Tailwind CSS, React Flow, Recharts, and Lucide for the investigator interface. Django and Python provide the API, evidence validation, analysis orchestration, persistence, reporting, integrity checks, and custody logic. TShark/Wireshark supplies packet dissection, while Zeek supplies structured network-security logs. Scapy supports packet handling and Scikit-learn provides the small Random Forest anomaly prototype alongside deterministic explainable scoring. PostgreSQL is the case system of record. In Supabase mode, Auth, Postgres, private Storage, Realtime, and PGMQ provide the managed data plane. Docker Compose and PowerShell scripts start and validate the local, controlled-LAN, Supabase, and production-gated profiles. ReportLab and PyPDF generate and verify submission artifacts.

## Challenges / Risks in Implementing the Final Prototype

The main risks are forensic privacy, model validity, scale, operational security, and legal interpretation. Packet captures may contain sensitive data, so evidence must remain authorized, encrypted, access-controlled, and excluded from screenshots or public repositories. Encrypted-traffic analysis is limited to metadata and behavior; NETRA cannot recover TLS content. The current detection benchmark has six cases and produced three false positives, while the Random Forest model has only six training rows, so neither supports production accuracy claims. Large captures require broader performance tests and hardened asynchronous workers. Replay and native sensor capture remain advanced, environment-dependent paths. Supabase RLS, Data API exposure, secret rotation, RBAC, SIEM signing, and least-privilege policies require review before shared deployment. Finally, hash and custody mechanisms support legal review but do not guarantee court admissibility, which depends on jurisdiction, procedures, examiner practice, and authority approval.

## Possible Outcome of Your Work

NETRA could give authorized investigators a more coherent path from a packet capture to a reviewable case package. Instead of manually transferring context between a packet viewer, security logs, spreadsheets, screenshots, notes, and report templates, an investigator could keep evidence identity, sessions, suspicious findings, explanations, visual relationships, exports, and custody actions connected to one case. This may reduce avoidable context switching, improve traceability, and make technical network evidence easier for supervisors or partner teams to review. Explainable rules and feature-based anomaly hypotheses can support human judgment without presenting an opaque score as fact. Encrypted storage, integrity verification, and custody history can strengthen evidence-handling discipline. The intended longer-term outcome is a field-tested, security-hardened platform for controlled cybercrime workflows, but measurable time savings, accuracy, usability, operational reliability, and legal acceptance must be established through broader benchmarks and investigator pilots.

## Accomplishments to Date

The repository contains a working React investigation console and Django analysis backend. The primary stored PCAP/PCAPNG workflow is implemented with capture validation, hashing, encryption, private artifact storage, TShark parsing, Zeek integration, session reconstruction, packet and protocol evidence, alert rules, anomaly records, graph and timeline data, case persistence, reports, exports, integrity checks, and a hash-linked custody ledger [CLM-001 to CLM-006]. Current health checks report TShark and Zeek as available. A six-case detection benchmark recorded five expected attack detections, no false negatives, three false positives, precision 0.625, recall 1.0, and F1 0.7692; these are engineering smoke results, not field certification [CLM-008]. A RandomForestClassifier artifact is active, but it was trained on only six rows and is presented strictly as an explainable prototype [CLM-007]. Legal-readiness validation passed its technical checklist for one test case, including manifest, report, export, access log, and three verified custody events [CLM-006]. Deployment remains production-gated, and replay, live capture, large-PCAP scale, security policy review, and legal acceptance remain roadmap work.
